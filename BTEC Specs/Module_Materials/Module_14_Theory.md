# Module 14: Project Planning & Management
## Theory & Teaching Notes

**Module Duration:** 2-3 hours  
**Level:** Professional Skills (Across Levels)

---

## Learning Objectives

By the end of this module, students will be able to:
- Understand project planning principles
- Define project scope and requirements
- Break down projects into tasks
- Create project schedules
- Identify dependencies and critical paths
- Track progress and manage risks

---

## 1. Project Planning Basics

### Why Plan?

**Planning helps you:**
- Organize work
- Manage time
- Allocate resources
- Identify problems early
- Ensure completion
- Meet deadlines

**Think of it like:** A roadmap for your project - you know where you're going and how to get there.

### Project Lifecycle

**Typical stages:**

1. **Initiation**
   - Define project
   - Get approval
   - Set up

2. **Planning**
   - Detailed planning
   - Schedule
   - Resources

3. **Execution**
   - Do the work
   - Monitor progress
   - Manage changes

4. **Monitoring & Control**
   - Track progress
   - Manage issues
   - Adjust as needed

5. **Closure**
   - Complete work
   - Document
   - Review

## 2. Defining Project Scope

### Project Aims

**Aims** = What you want to achieve (high-level goals)

**Example:**
- "Build a robot arm that can pick and place objects"
- "Create an automated assembly system"
- "Design a robot cell for sorting"

**Characteristics:**
- Broad
- High-level
- Overall goal

### Objectives

**Objectives** = Specific, measurable goals

**Example:**
- "Robot can pick objects up to 500g"
- "Repeatability of ±1mm"
- "Cycle time under 5 seconds"

**Characteristics:**
- Specific
- Measurable
- Achievable

### Requirements

**Requirements** = What the system must do

**Types:**

1. **Functional Requirements**
   - What it must do
   - Features needed
   - Capabilities required

2. **Performance Requirements**
   - How well it must do it
   - Speed, accuracy, etc.
   - Quality standards

3. **Constraints**
   - Limitations
   - Budget, time, resources
   - Restrictions

### Success Criteria

**Success criteria** = How you know project succeeded

**Example:**
- All requirements met
- Within budget
- On time
- Quality acceptable

3. **Organize logically**

   - Identify dependencies

**Testing tasks:**

**Documentation tasks:**
- Writing

## 4. Scheduling

### Time Estimation

**How to estimate:**

1. **Break task into steps**
   - List all steps
   - Estimate each step
   - Add up

2. **Consider factors**
   - Complexity
   - Experience
   - Resources available
   - Dependencies

3. **Add buffer**
   - Unexpected problems
   - Learning curve
   - Contingency

**Tips:**
- Be realistic
- Consider worst case
- Ask experienced people
- Review similar projects

### Creating Timeline

**Tools:**

1. **Gantt Chart**
   - Visual timeline
   - Shows tasks and durations
   - Shows dependencies
   - Easy to understand

2. **Simple Timeline**
   - List tasks with dates
   - Sequential order
   - Simple format

3. **Milestone Chart**
   - Key dates
   - Major achievements
   - Progress markers

### Dependencies

**What are dependencies?**

**Dependency** = One task must be completed before another can start

**Types:**

1. **Finish-to-Start**
   - Task A must finish before Task B starts
   - Most common

2. **Start-to-Start**
   - Task A must start before Task B starts
   - Parallel work

3. **Finish-to-Finish**
   - Task A must finish before Task B finishes
   - Coordinated completion

**Example:**
- "Assemble robot" must finish before "Test robot" starts
- **Dependency:** Assembly → Testing

### Critical Path

**What is critical path?**

**Critical path** = Longest sequence of dependent tasks

**Why it matters:**
- Determines project duration
- Any delay on critical path delays project
- Focus attention here

**How to find:**
1. List all tasks
2. Identify dependencies
3. Find longest path
4. That's critical path

**Example:**
- Path 1: Design (2 days) → Build (5 days) → Test (2 days) = 9 days
- Path 2: Order parts (3 days) → Receive (2 days) = 5 days
- **Critical path:** Path 1 (9 days)

## 5. Project Management

### Progress Tracking

**Why track progress:**
- Know where you are
- Identify problems early
- Adjust if needed
- Report status

**How to track:**

1. **Regular updates**
   - Daily or weekly
   - Update status
   - Record progress

2. **Status indicators**
   - Not started
   - In progress
   - Complete
   - Blocked

3. **Percent complete**
   - Estimate completion
   - Track progress
   - Visualize

### Risk Management

**What is risk?**

**Risk** = Something that could go wrong

**Risk management process:**

1. **Identify risks**
   - What could go wrong?
   - List possibilities

2. **Assess risks**
   - Likelihood (how likely?)
   - Impact (how bad?)
   - Risk level

3. **Mitigate risks**
   - How to reduce likelihood?
   - How to reduce impact?
   - Plan responses

4. **Monitor risks**
   - Watch for risks
   - Update assessment
   - Take action if needed

**Example risk:**
- **Risk:** Parts delayed
- **Likelihood:** Medium
- **Impact:** High (delays project)
- **Mitigation:** Order early, have backup supplier

### Communication

**Why communicate:**
- Keep stakeholders informed
- Coordinate team
- Share information
- Get help

**Who to communicate with:**
- Team members
- Supervisor/instructor
- Stakeholders
- Others involved

**What to communicate:**
- Progress updates
- Problems encountered
- Decisions made
- Changes needed

**How often:**
- Regular updates
- When problems occur
- When decisions needed
- As appropriate

### Key Concepts

1. **Planning** organizes work and ensures completion

3. **Task breakdown** divides work into manageable pieces
4. **Scheduling** plans when work happens
5. **Dependencies** show task relationships
6. **Critical path** determines project duration
7. **Management** tracks progress and manages risks
